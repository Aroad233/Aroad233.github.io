对laravel记录ip的一些思考

前言：前段时间开发的时候遇到了需要存储记录ip的功能点，然后便去laravel社区搜了搜发现$request->ip()所获得的ip可能并不是真实的ip…而后就有了这篇文章。

php获得真实用户ip

laravel中自带的ip函数

​ $request->ip()指向函数$request->getClientIp()而通过查看$request->getClientIp()源码可以得知我们获得的是REMOTE_ADDR

我们继续查看getClientIp() 的备注

大概解释一下就是getClientIp也可以获得请求头中的X-Forwarded-For当设置了函数setTrustedProxies()（设置信任代理）后。X-Forwarded-For的值是逗号跟空格分隔的ip地址列表，第一个是原始客户端，以及后续传递请求的每一个代理获得的ip地址

如果反向代理使用跟X-Forwarded-For不同的头名称，那么用Request::setTrustedProxies()方法将配置参数$trustedHeaderSet改一下。

$_SERVER[‘HTTP_X_FORWARDED_FOR’] 是有标准定义，用来识别经过 HTTP 代理后的客户端 IP 地址，格式：clientip,proxy1,proxy2。详细解释见 wiki for x-forwarded-for

ip的在mysql中的存储

再来谈一谈，ip再mysql当中的存储问题

ipv4在MySQL中的存储

​ 最开始以为ipv4在mysql中存储可以直接使用varchar，但在看高性能mysql的时候发现可以使用int通过inet_aton与inet_ntoa来转换存储，而且int在查询、运算索引等方面的效率上比varchar要高出很多。 int只占4字节 varchar则会占15字节 一旦表数据行数到了1亿，那么要多用1.1G存储空间。

​ 而后发现在php中可以使用ip2long跟long2ip来进行ipv4 string与int类型的转换。

​ 但ip2long存在一个问题在不同系统或不同平台转换所获得的结果并不同比如64位机器与32位机器会超出int的范围。所以有以下几个解决方案：

  1. 使用mysql函数inet_aton跟inet_ntoa来处理存入的ip
  2. 将ip的字段改为bigint类型
  3. 使用sprint('%u',ip2long($ip))转换
  4. 自己手写ip2long()

​ 使用bigint(8 字节 64 位) 存储 ip2long 是完完全全的浪费。而使用inet_aton与inet_ntoa总感觉不应该在mysql中操作转换。而后继续查阅资料发现，可以通过使用sprint('%u',ip2long($ip))或者自己手写转换的ip2long()，并将mysql数据库中的ip字段类型设置为int unsigned即可解决此类事情。

为何ip2long会有不用的结果

因为32位机器上ip2long int类型会溢出

ipv6在mysql中的存储

​ ipv4可以使用int存储，但ipv6就不能这样了，ipv6实际是16字节的整数，但laravel官方是直接将migration表中$table->ipAddress('visitor')对应成mysql中的varchar(45)，而我在其他地方发现可以使用varbinary(16)来进行存储

mysql中也自带了INET6_ATON 和INET6_NTOA来供给我们进行转换

而php则无法使用ip2long进行ipv6的转换了 因为ipv6是128位的，所以就需要自己手写方案解决了
